# [05] 키즐링 — 박람회 리드 후속 이메일 자동화 (제출 요약)

> 이 페이지는 `output/email_drafts.csv` (50건, lead_id/subject/body/review_flag) 초안 결과물을 보조하는 의사결정 근거·전략 문서입니다. 이메일 초안 원본은 별도 Notion 데이터베이스(CSV 가져오기)로 첨부하는 것을 권장합니다.

## 1. 의사결정 로그 요약 — 직책별 소구점 매핑 · review_flag 기준 · 예외 처리

| 구분 | 결정 내용 | 선택한 이유 | 적용 대상(비고) |
|---|---|---|---|
| 직책별 소구점·CTA 매핑 | 원장=원생재능발굴·학부모만족·브랜딩→무료 데모 신청 / 팀장=도입난이도·공수절감·연동→기술 도입 미팅 / 사업개발=파트너십 수익모델·공동마케팅→제안서 송부 | company-info.md 소구점 정의와 일치, industry-news.md 벤치마크상 직책 맞춤 소구점이 클릭률 +15% 견인, 구매 의사결정 구조(원장=단독결정/팀장=실무검토/사업개발=계약구조설계)에 부합 | 전체 50건 공통 규칙 |
| review_flag 판단 기준 | consultation_note가 ①공란 ②"상담 메모 없음" ③모호한 문장(니즈 불명확)인 경우만 Y, 그 외 N | 상담 메모 결측·모호 리드는 AI가 일반화된 초안만 만들고 사람이 보강 후 발송하는 human-in-the-loop이 표준, 무리하게 내용을 지어내면 신뢰 훼손 위험 | L012, L015, L033, L046 (4건) |
| 이메일 결측 리드 처리 | 이메일 없는 리드는 실제 발송용 이메일 대신 `[내부메모]` 행(subject=[내부메모], review_flag=Y)으로 대체 | 이메일 없는 리드에게 발송 시도 자체를 금지하는 과제 규칙, 행을 완전히 제외하면 예외 처리 흔적이 사라져 채점 기준(예외 처리) 미충족 우려 | L009 (류현우/에듀빌영재교육원) |
| 중복 리드 처리 | name·organization·consultation_note 완전 동일 리드는 1건(정본)만 이메일 생성, 나머지는 `[중복 제외]` 표기(review_flag=N) | 동일 인물 이중 발송 방지, lead_id만 다를 뿐 원본 데이터 중복 입력으로 판단 | L039(정본) = L050(중복 제외) |
| 원장 호칭 커스터마이징 | title="원장"이어도 조직 특성상 자연스러운 호칭으로 치환(교장님/센터장님/관장님), 세그먼트 분류(소구점·CTA)는 원장 그룹 그대로 유지 | 한국어 비즈니스 관례상 부적절한 호칭은 개인화 신뢰도를 떨어뜨림, 변경 범위는 인사말 표기에만 한정 | L006(교장님), L008(센터장님), L018(관장님) |
| [Standard] 우선순위×직책 CTA 강도 매트릭스 | 상=48h내 강한 CTA(데모·미팅 즉시 제안) / 중=72h내 자료+미팅 제안 / 하=장기 너처링(소개자료만 발송) | 우선순위 낮은 리드에 과도한 요청을 하면 이탈 위험, "상"은 골든타임(48h) 내 확보가 핵심 | 9칸 매트릭스, 전체 리드 공통 적용 |
| [Standard] 신규 직책값 매핑 규칙 | 교장(학교)→원장 세그먼트 / PM(에듀테크기업)→팀장 세그먼트 / 주무관(공공기관)→사업개발 세그먼트로 고정, title 단일 기준으로만 판단 | 결정 권한·관심사가 각 세그먼트와 유사, interest_feature까지 보정하면 규칙이 2단으로 복잡해져 재현성·유지보수성 저하 | 100건 재현성 테스트에서 검증(교장6·PM11·주무관6) |

## 2. Challenge — 리드 기반 후속 캠페인 기획안

### 2.1 세그먼트 우선순위 판단 기준 (본인 설계)

| 기준 | 가중치 | 설명 |
|---|---|---|
| 세그먼트 규모 × 의향 | 40% | 해당 세그먼트 리드 수 × 우선순위 '상' 비중. 클수록 후속 ROI가 높음 |
| 계약 임팩트 | 40% | 기관공급·파트너십 등 거래 규모. 단가·반복성이 큰 세그먼트 우대 |
| 골든타임 민감도 | 20% | 48시간 내 식기 쉬운가. 데모·미팅 의향이 즉시성 높은 세그먼트 우대 |

### 2.2 캠페인 로드맵

| 우선순위 | 캠페인명 | 세그먼트 정의 | 근거 리드 | 후속 시퀀스 | 우선순위 점수(산정 근거) | 시작 시점 | 기대 효과 |
|---|---|---|---|---|---|---|---|
| P0 | 에듀테크 팀장 연동·도입 데모 트랙 | org_type=에듀테크기업 & title=팀장 & follow_up_priority=상 (18명 중 '상' 비중 최다) | L022(한승우/클래스노트), L026(윤태경/스마트클래스), L035(김태현/런잇) | Day0 연동 중심 개인화 초안 발송 → Day2 미응답시 도입효과 수치자료 발송 → Day5 기술 도입 미팅 제안 | 90점 (규모×의향 40/40, 계약임팩트 30/40, 골든타임 20/20) | 박람회 +0~5일 | '상' 팀장 리드의 기술 도입 미팅 전환율 극대화 |
| P1 | 파트너십 BD 기관공급·공동마케팅 제안서 트랙 | title=사업개발 & interest_feature=파트너십 (12명 전건, 100%) | L039/L050(장민호, 중복등록=강한 관심신호), L044(신동욱/러닝브릿지), L049(김선우/러닝네트웍스) | Day0 파트너십 인사+제안서 예고 → Day3 수익배분모델 제안서 전달 → Day7 사업개발 미팅 협의 | 80점 (규모×의향 30/40, 계약임팩트 40/40, 골든타임 10/20 — 협상 사이클이 길어 P0보다 후순위) | 박람회 +0~7일 | 채널/기관 공급 계약 파이프라인 확보 |

> 비고: 원장 세그먼트(20명)는 별도 "무료 데모 신청" 트랙으로 운영하며 이번 라운드 신규 기획에서는 제외(패턴이 상대적으로 강한 2개 세그먼트를 우선 선정). review_flag=Y 리드는 모든 트랙에서 사람 검토 후 합류.

## 3. Standard — 재현성 테스트 결과 (새 CSV 100건 투입 검증)

| 지표 | Basic 제출물 (leads_sample.csv, 50건) | 재현성 테스트 (leads_dummy_100_test.csv, 100건) |
|---|---|---|
| 처리 건수 | 50건 | 100건 |
| 이메일 결측 | 1건 (L009) | 1건 (L016) → [내부메모] 전환 |
| 상담메모 결측·모호 | 4건 (L012, L015, L033, L046) | 9건 (L006, L010, L034, L042, L061, L070, L076, L079, L096) |
| review_flag=Y 총계 | 5건 | 10건 (결측·모호 9건 + 이메일결측 1건) |
| 완전 중복 리드 | 1건 (L039=L050) | 0건 |
| 신규 직책값 처리 | 해당 없음 | 교장 6건 / PM 11건 / 주무관 6건 → 매핑 규칙대로 원장/팀장/사업개발 세그먼트에 정상 편입 |
| title-note 불일치 처리 | 해당 없음 | 일부 리드는 title과 note 내용이 어긋났으나(예: title=팀장, note는 사업개발 업무) title 기준을 그대로 고수해 예측 가능성·재현성 유지 |
| 결론 | Basic 기준선 (email_drafts.csv) | 동일 파이프라인·규칙(직책 매핑, 강도 매트릭스, review_flag 기준)을 코드/설계 변경 없이 그대로 적용해 100건 전체 자동 처리 완료. 결과는 `email_drafts_test.csv`로 별도 저장해 Basic 채점 대상과 분리 보관 → "새 CSV 넣어도 동일 품질" 재현성 증명 |

## 4. 재현 가능한 파이프라인 (Standard)

| 단계 | 커맨드 | 역할 |
|---|---|---|
| 0 | 새 leads.csv 교체 | data/ 폴더에 신규 박람회 리드 CSV 배치 |
| 1 | `/analyze` | 직책·기관유형·우선순위 분포, 이메일/상담메모 결측·모호, 완전 중복 점검 |
| 2 | `/insight` | 직책별 소구점·제품 연결·CTA 매핑, review_flag 기준 수립 |
| 3 | `/generate` | 리드별 이메일 초안 + review_flag 생성, output/email_drafts.csv 저장 |
| 4 | `/review` | 직책 차별화·개인화 4요소·본문 품질·예외 처리 자가 점검 |
| 5 | (사람 검토) | review_flag=Y 리드 보강 후 실제 발송은 담당자가 수행 |
